Encode bitrate is the target passed to the GStreamer encoder. WebRTC max is the ceiling for the GCC congestion controller (should be ≥ encode bitrate). WebRTC start is the initial probe rate (~50% of max is a safe default). WebRTC min is the floor GCC won’t drop below.

Setup guide

Step 1 — Create a layout in MavSphere

In the MavSphere UI, create a railway layout and note its numeric Layout ID. Enter that same value in the Layout ID field above.

Step 2 — Configure backend access

Enter the backend HTTP URL, backend WebSocket URL, and your MavSphere login credentials. The agent logs into the backend, fetches train and topology data, and subscribes to control messages.

Step 3 — Configure DCC-EX

In DCC-EX Port, enter either:

  • /dev/ttyUSB0 or similar for a direct serial connection
  • host:port for a TCP connection, for example 192.168.1.50:2560

Do not include http://, ws://, or any path. Use only a serial device path or a plain host:port value.

Step 4 — Configure MQTT

Set the MQTT broker URL used by local layout devices such as sensor nodes and signal controllers. In most deployments this will be the local Mosquitto broker, for example tcp://localhost:1883.

Step 5 — Configure trains in MavSphere

Trains are managed in the MavSphere UI, not in this page. For each train, configure its DCC address, slug, display name, guarding signal, and start block. The agent fetches the train list from the backend at startup and displays it here as read-only.

Step 6 — Configure cameras

Add any local camera devices you want the agent to publish through Janus WebRTC. Enter a friendly Camera name; the agent generates the internal camera ID automatically from that name. Use Scan for cameras to find capture devices and supported modes. Typical Linux device paths are /dev/video0 and /dev/video2. Use camera type ONBOARD only when a camera should be tied to a specific train. The train field can be selected from the detected train list or typed manually if the train has not been created yet.

Step 7 — Start and verify

Start the agent and watch the Status panel above. Backend STOMP and DCC-EX should both go green. If cameras are configured, the backend can then request streams from the agent via Janus.

Video quality quick-reference

Pi (Raspberry Pi 4/5): codec = H.264, encoder = auto or v4l2h264enc, profile = constrained-baseline, encode bitrate = 1.5–2 Mbps. The Pi hardware encoder disables GCC congestion control; bitrate is fixed at whatever you set.

PC / Ubuntu desktop: codec = H.264, encoder = auto (resolves to x264enc), profile = constrained-baseline, encode bitrate = 2–4 Mbps. GCC congestion control remains active; the stream adapts to available bandwidth up to WebRTC max.

Check the Live Stream Status panel while a stream is running to confirm what is actually being used. The agent logs also print [stream] starting gst-launch … with the full pipeline on each start.

Troubleshooting

If DCC-EX does not connect:

  • For TCP, confirm the value is plain host:port and not a URL
  • If using TCP, confirm the DCC-EX device or bridge is actually listening on the configured port
  • For serial, confirm the device path exists and is accessible inside the container
  • Check the agent logs for connection errors

If video does not start:

  • Check the Live Stream Status panel — if codec shows — no stream is running
  • Check agent logs for [stream] starting gst-launch and any errors that follow
  • Confirm the camera device exists at the configured /dev/videoX path
  • Try a lower resolution or frame rate
  • Check Janus connectivity and ICE/TURN configuration if WebRTC sessions fail
  • Run gst-inspect-1.0 x264enc to confirm the encoder plugin is installed
